## 2. База данных (database.py)
- Функции:

    - get_db() – возвращает соединение с game.db, устанавливает row_factory = sqlite3.Row для доступа к полям по имени.

    - now_msk() – текущее время в московском часовом поясе.

    - hash_password(password) / verify_password(stored, provided) – хеширование и проверка паролей (PBKDF2 SHA-256 с солью).

    - init_db() – создаёт все таблицы, если их нет, заполняет начальными данными (60 карт, квесты, достижения, товары магазина). Также до-добавляет столбцы и генерирует nfc_hash для карт.

- Основные таблицы (сущности):

    - users – учётные записи игроков (имя, пароль, email, стихия, монеты, обучение, аватар и т.д.).

    - pending_registrations – временные записи для двухэтапной регистрации.

    - user_cards – коллекция карт у пользователей (связь M:N с таблицей cards), количество и уровень.

    - cards – справочник карт (название, тип действия, стихия, базовая сила, редкость, nfc_hash).

    - story_progress – прогресс сюжета (глава, индекс боя).

    - daily_cases – дата последнего кейса и стрик.

    - daily_quests / user_quests – ежедневные задания и прогресс.

    - achievements / user_achievements – достижения и их получение.

    - user_stats – победы/поражения/урон.

    - showcase – витрина (3 слота с картами).

    - pvp_queue – очередь поиска боя.

    - pvp_matches – PvP-матчи.

    - pvp_rating – рейтинг ELO.

    - trade_offers – торговые предложения.

    - friends – запросы дружбы (со статусами).

    - friend_invites – токены приглашений.

    - shop_products, shop_orders, shop_order_items, shop_payments – магазин.

## 3. Модели данных
```models/user.py``` – класс ```User```
- Инкапсулирует данные пользователя: id, имя, стихия, здоровье, аватар, флаги (trained, cursed, allow_duels).

- Ключевые методы:

    - ```get_element_multiplier()``` – вычисляет модификатор урона: 1.0 + 0.2 (если обучен) – 0.2 (если проклят).

    - ```change_nickname()```, ```set_element()```, ```set_avatar()``` – изменяют поля с проверкой ограничений (раз в месяц для ника и стихии) и записью в БД.

    - ```apply_curse(hours)``` – активирует проклятие на время.

    - ```health``` – свойство с ограничением 0–100.

```models/card.py``` – классы ```Card``` и ```ChainLightning```
- ```Card```: базовый класс, поля id, name, action_type, element, base_value, mana_cost, description, rarity.

- Метод ```use(user, battle)``` – выполняет действие в бою:

    - Атака: вычисляет множитель стихии (```get_damage_multiplier```) и бонус игрока, наносит урон.

    - Защита: добавляет щит.

    - Лечение: восстанавливает здоровье до 100.

    - Пропуск: заставляет противника пропустить ход.

- ```ChainLightning(Card)``` – переопределяет use рекурсивной атакой: повторяет удары с уменьшением урона на 5, пока здоровье врага > 0.

```battle.py``` – классы ```Battle``` и ```PVPBattle```
- ```Battle``` (сюжетный бой):

    - Создаётся с игроком, врагом (здоровье, карты, стихия) и копией пула карт игрока.

    - Разделяет карты игрока по группам (attack, defense, heal, skip) для отображения.

    - ```apply_player_card(card_id)``` – проверяет ход, применяет карту, записывает историю, затем запускает enemy_turn.

    - ```enemy_turn()``` – ИИ: фильтрует карты (не лечиться при здоровье >80%, не защищаться при наличии щита), выбирает случайную из доступных, применяет.

    - ```apply_boost(card)``` – NFC-усиление (множитель ×1.5) один раз за бой.

    - Вспомогательные методы: расчёт урона с учётом щитов, лечения.

- ```PVPBattle```:

  - Аналогично, но для двух игроков: поочерёдные ходы, проверка skip, определение победителя.

  - ```apply_card(player_id, card_id)``` – получает id игрока через API, модифицирует пул карт.

- ```GET /logout``` – очистка сессии.

## 6. Система шаблонов

- ```base_game.html``` – публичный шаблон (для гостей). Содержит шапку с логотипом и навигацией (Как играть, FAQ, Магазин, Контакты), кнопки Войти/Регистрация (или Игровая панель/Выйти, если пользователь авторизован). Футер с политикой конфиденциальности.

- ```base_private.html``` – для авторизованных страниц. Шапка с аватаркой и именем пользователя (из переменной ```current_user```), ссылки «Главная» и «Выйти». Все внутренние страницы (дашборд, профиль, инвентарь, бой и т.д.) расширяют его.

- Шаблоны активно используют Jinja2: циклы, условия, фильтры (например, ```selectattr('slot')```).

- Данные в шаблоны передаются из маршрутов в виде переменных (например, ```current_user```, ```cards```, ```battle```). Везде используется ```current_user``` для получения данных игрока.


## 7. Механика NFC
- <b>Web NFC API</b> (браузерный JS) – работает только в Chrome на Android при HTTPS.

- Для получения карты: ```nfc.js``` считывает текстовые данные с метки (ожидается nfc_hash) и отправляет на ```/api/nfc_scan```.

- Для усиления: ```nfc_boost.js``` (в бою) считывает серийный номер метки и отправляет на ```/api/nfc_boost```.

Сервер (```nfc.py```) проверяет, не истекли ли 3 дня, ищет карту по хешу/серийному номеру, обновляет ```last_nfc/last_nfc_boost```.


## 8. Интеграция платежей (ЮKassa)
- <b>Сервис-класс</b> ```YookassaService``` (в ```services/yookassa_service.py```):

    - Инициализируется ключами из ```.env```.

    - ```create_payment()``` – отправляет запрос в API ЮKassa, возвращает URL для редиректа.

    - ```get_payment()``` – проверяет статус.

- <b>Жизненный цикл заказа</b>:

  1. Оформление → заказ в статусе new.
  2. Создание платежа → редирект на ЮKassa.
  3. После оплаты → возврат на ```/payment/result```, где статус проверяется через API; также работает вебхук для надёжности.
  4. При успешной оплате: заказ получает статус paid, пользователю начисляются цифровые товары (монеты), отправляется чек на email.
    - Чек формируется простым текстом (можно расширить до HTML/PDF).
